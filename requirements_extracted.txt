Innovation Initiative Management ApplicationDesign Document & Implementation RequirementsVersion 1.0  |  April 20261. Executive SummaryThis document defines the design and implementation requirements for the Innovation Initiative Management (IIM) application — a lightweight, purpose-built platform for tracking, managing, and prioritizing innovation projects and tasks within your team.Inspired by the simplicity and speed of tools like Linear, IIM replaces the current spreadsheet-based workflow with a structured, AI-assisted application. It supports multiple roadmap views, role-based access control, open project submission, and AI-powered recommendations — while leaving the door open for future integration with GitHub Projects or Jira.The initial deployment will use JSON files as the data store — no database server required. All storage is abstracted behind a dedicated data access layer so the backend can be swapped to any database (PostgreSQL, MySQL, MongoDB, or others) in the future with no changes to the UI or business logic.2. Guiding PrinciplesLightweight over heavy: Optimize for speed of use and ease of daily adoption, not feature completeness for edge cases.Data fields over labels: Use structured fields to power filtering, AI analysis, and reporting — not free-form tags.AI-assisted, not AI-dependent: AI recommendations are advisory; humans make final decisions.Open submission, controlled promotion: Anyone can submit an idea; only authorized users can review and promote it.Storage-agnostic: All data access goes through an abstraction layer. The initial implementation uses JSON files; any database can be substituted later by replacing only that layer.Integration-ready: Architecture anticipates GitHub Projects and Jira integration without requiring it on Day 1.3. No page, component, or API route ever reads or writes storage directly. This is the architectural decision that keeps the database choice open.File Structurelib/db/ — the data access layer; one file per entity (projects.ts, tasks.ts, ideas.ts, users.ts, notifications.ts, decisions.ts)lib/db/types.ts — shared TypeScript interfaces for all entities (Project, Task, Idea, User, etc.)lib/db/index.ts — re-exports all repositories; the only import other files needdata/ — the JSON store; one file per entity (projects.json, tasks.json, etc.); gitignored in production, seeded from sample data in developmentData Access Layer PatternEach entity file exports a plain object (repository) with async functions. Example for projects:export const ProjectRepository = {getAll(): Promise<Project[]>getById(id: string): Promise<Project | null>create(data: CreateProjectInput): Promise<Project>update(id: string, data: Partial<Project>): Promise<Project>delete(id: string): Promise<void>}The JSON implementation of each function reads the relevant .json file, applies the operation in memory, and writes the file back. When a database is introduced, the function bodies are replaced with database queries — the function signatures, and everything that calls them, stay identical.JSON File ConventionsEach JSON file contains a top-level array of records: projects.json = [ { ...project }, ... ]IDs are strings in application-defined formats (YYYY-NNN for projects, YY-NNNN for tasks)Timestamps stored as ISO 8601 strings (e.g., '2026-04-23T14:30:00Z')All reads load the full file into memory; suitable for hundreds of records — if the dataset grows to thousands, a database swap is the right triggerWrite operations use atomic file writes (write to a temp file, then rename) to prevent corruption on concurrent savesThe data/ folder is excluded from version control via .gitignore; a data/seed/ folder contains sample data for developmentSwapping to a Database LaterWhen the team selects a database, the migration path is:Choose an ORM or query builder (Prisma, Drizzle, Mongoose, etc.) and install itTranslate the TypeScript interfaces in lib/db/types.ts into the database schemaRewrite the function bodies in each lib/db/*.ts file to use database queriesRun a one-time migration script to import the existing JSON files into the databaseDeploy — no changes needed to any page, component, API route, or business logic3.4 Core ModulesProject Repository — master list of all innovation projectsTask Manager — tasks linked to projects with resource assignmentRoadmap Views — Timeline and Kanban visualizationsReview Pages — Projects, Tasks, My Tasks dashboardsAI Advisor — complexity estimation, priority recommendation, overlap detectionIdea Submission Portal — public-facing form, no login requiredAdmin Console — user management, field customization, template management4. A dependency means the dependent project cannot meaningfully start or complete a specific phase until the upstream project reaches a defined state. Dependencies are visible on both the project record and the Timeline roadmap view.On any project form, a 'Depends On' field allows selecting one or more upstream projects from the project repositoryDependency type options: Blocks Start (upstream must complete before this can begin) or Blocks Phase (upstream must reach a specified phase)Timeline view: dependency relationships rendered as connector arrows between project bars; arrow color indicates status of the upstream project (green = clear, orange = at risk, red = blocked)Dependency warning banner: if an upstream project is Blocked, Delayed, or On Hold, a warning appears on the dependent project record and in the Projects page health indicatorCircular dependency detection: the system prevents A → B → A dependency chains at save timeDependency chain view: a collapsible tree on each project record showing the full upstream dependency chainAI Priority Recommendation accounts for dependencies: projects with no upstream blockers are ranked higher when all else is equalData model: ProjectDependency join table with fields: dependent_project_id, upstream_project_id, dependency_type, required_phase (nullable)PPTX export: Timeline slide includes dependency arrows; a standalone Dependency Map slide shows the full graph5.11 Decision & Change LogA structured log of significant decisions made on each project — why scope changed, why a target date moved, why a project was reprioritized. Gives an at-a-glance signal that 'In Progress' does not necessarily mean 'on track'.Scoring LogicGreen (Healthy): fewer than 20% of tasks are blocked or overdue; target date is more than 2 weeks away or no target date set; at least one task updated in the last 7 daysYellow (At Risk): 20-40% of tasks are blocked or overdue; OR target date is within 2 weeks with more than 30% of tasks still open; OR no task activity in 14+ daysRed (Critical): more than 40% of tasks are blocked or overdue; OR target date has passed and project is not Completed; OR project status is Blocked; OR an upstream dependency is RedThresholds are configurable by Admin in the Admin ConsoleScore is recalculated on every task update and project field change; stored on the project record for query performanceHealth score history tracked: a sparkline on the project detail page shows how the score has changed over the last 30 daysHealth score used as an input to the AI Priority Recommendation — Red projects with high priority are surfaced prominentlyBlocked / At-Risk PPTX slide pulls directly from health score to identify projects to highlight5.14 Document & Repository LinksEach project can have an unlimited number of labeled links to external documents, repositories, and tools.
