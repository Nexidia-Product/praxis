# PPTX export branding

The PPTX export reads a swappable template from this folder and inherits
its theme (colors, typeface) automatically. With pre-rendered cover and
content background images, the deck also picks up the template's
graphical identity (gradient cover, brand mark on content pages).

## Files

| File | Required? | Purpose |
| --- | --- | --- |
| `template.pptx` | Yes | Source of theme colors, typeface, and (via `prepare:branding`) the rasterized layouts. |
| `cover.png` | Optional | Full-bleed background for the title slide. Generated by `npm run prepare:branding`. |
| `content.png` | Optional | Full-bleed background for every native content slide. Generated by `npm run prepare:branding`. |

If the template is missing, the export falls back to the
`BrandingConfig` stored in the `settings` row and the previous
code-drawn slide layouts. You won't lose the export, just the template's
visual identity.

## Replacing the template

1. Drop the new template at `public/branding/template.pptx` (or set
   `PRAXIS_PPTX_TEMPLATE_PATH` to its absolute path).
2. Run `npm run prepare:branding` to regenerate `cover.png` and
   `content.png` from the new template.
3. Restart the dev server (or just trigger a new export — the runtime
   parses the theme on every request, cached by mtime).

## Public exposure note

Files in `public/` are served by Next.js at their root path, so the
template assets are reachable at `/branding/template.pptx` etc.
Brand templates are not secret material; this trade-off keeps the
deploy bundle simple. If you need to hide them, configure
`outputFileTracingIncludes` in `next.config.js` and move the folder
back outside `public/`.

## Layout selection

`prepare:branding` reads the template's first layout as the cover and
its fifth layout as the content background. Override with environment
variables when a future template reorders its layouts:

```sh
PRAXIS_BRANDING_COVER_PAGE=2 PRAXIS_BRANDING_CONTENT_PAGE=6 npm run prepare:branding
```

## Requirements for `prepare:branding`

LibreOffice and `pdftoppm` (poppler-utils) on PATH. The runtime export
route does NOT need either — it only reads the rasterized PNGs.
